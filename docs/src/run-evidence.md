# Run Evidence Architecture

Run evidence is hpc-compose's local, durable account of what was submitted and
which user-authored annotations were later attached to that submission. It is
an additive protocol beside `SubmissionRecord`: existing job tracking remains
the compatibility source for scheduler operations, while evidence gives new
code a stricter foundation for provenance and historical views.

This page is both an architecture decision record and a contributor contract.
It describes the invariants that later consumers must preserve and separates
them from the smaller set of integrations implemented today.

## Decision and scope

hpc-compose uses a **local-first run evidence protocol** with five parts:

- an immutable, schema-versioned `RunManifest`;
- an opaque, globally unique `run_id` that is distinct from the scheduler's
  site-local `job_id`;
- an immutable `InputsLock` describing the inputs known at submission time;
- a strictly locked, append-only stream of `RunEvent` records; and
- an atomically replaced, rebuildable `RunView` derived from the immutable
  documents and event stream.

No daemon, database, network service, or scheduler-side writer is required.
Files live in the compose-level metadata root and remain understandable and
recoverable with ordinary filesystem operations.

The current run-evidence schema version is `1` for the manifest, input lock,
events, and view. A shared version keeps the initial protocol easy to audit;
future versions can split only if their compatibility needs genuinely diverge.

### Current phase

The first phase integrates evidence with new submissions and with explicit
annotation mutations. Persisting a new tracked record attempts to create its
manifest, input lock, and initial materialized view. Adding a tracked note or
updating tags attempts to append a typed event under the event-stream lock and
refresh the view. Other mutable record fields do not yet have evidence-event
equivalents.

Legacy record persistence is still the phase-one commit boundary. If additive
evidence initialization or synchronization fails after that commit,
hpc-compose emits the coded `run_evidence_degraded` warning and keeps the
already-submitted or updated tracked operation successful. A later idempotent
record write or annotation update retries initialization and synchronization.
This makes degraded evidence visible without turning a successfully submitted
scheduler job into an ambiguous CLI failure.

Scheduler observation commands remain read-only. In particular, `status`,
`ps`, `watch`, `replay`, and other scheduler reads do not append observations
merely because a user inspected a job. This keeps reads free of hidden writes
and prevents polling frequency from becoming protocol history.

The initial read-only consumer is `hpc-compose experiment bundle`: it can
package evidence that is already present for the tracked run and preserves the
legacy bundle path when evidence is absent. Bundle export does not initialize,
repair, or append evidence, and it does not introduce a comparison or result
protocol.

The following are **not implemented by this phase**:

- experiment comparison or a generalized result/metric contract;
- support-bundle export or automatic redaction for sharing;
- workspace-expiry decisions;
- sweep active-window control;
- general parent/child lineage across runs; and
- sealed plan-to-submit receipts.

Those capabilities may consume this protocol later, but their interfaces and
semantics are deliberately not claimed here.

## Identity model

`run_id` and `job_id` answer different questions and must never be conflated:

| Identifier | Meaning | Properties |
| --- | --- | --- |
| `run_id` | hpc-compose identity for one tracked run | Opaque, globally unique, generated by hpc-compose, stable across local views |
| `job_id` | Scheduler identity returned for a submission | Site-local, scheduler-defined, useful for scheduler commands and compatibility lookups |

Code must treat both values as data, not unchecked path components. The
current on-disk layout remains keyed by the validated scheduler job ID for
compatibility with tracked records; the manifest's `run_id` is the durable
protocol identity. Future cross-cluster consumers should key on `run_id` and
carry scheduler/site identity separately instead of inventing compound IDs.

The compatibility layout still assumes that one compose metadata root does not
contain two distinct runs with the same scheduler job ID. A conflicting
initialization fails closed and preserves the first immutable documents. This
phase is therefore not a cross-cluster aggregation store; a future aggregation
layout must be keyed by `run_id` or an explicit cluster namespace.

## Evidence states

Optional or fallible observations use `Evidence<T>` rather than overloading
`null`, an empty value, or a swallowed error. An evidence value carries:

- a status that distinguishes an available value from missing, unsupported,
  or degraded evidence;
- an optional typed value when the status permits one; and
- an optional human-readable reason for non-available or qualified evidence.

The status/value combinations are validated at the protocol boundary. In
particular, an available observation has a value, while missing and unsupported
observations do not pretend that a measurement succeeded. Readers must handle
unknown future statuses conservatively through schema-version handling rather
than silently treating them as available.

## Submit-time documents

`RunManifest` records the run and scheduler identities, submission backend and
kind, submission time, configuration and bounded rendered-script digests,
submit-time provenance evidence, the SHA-256 identity of the serialized input
lock, and a SHA-256 binding to a frozen version-1 projection of the immutable
`SubmissionRecord` fields. Tags and notes are excluded from that projection
because their post-submit changes are represented by typed events. Freezing the
projection prevents a future additive record field from silently changing old
protocol identities. The manifest records digests rather than embedding
configuration or script bytes.

Each `InputsLock` entry names its input kind and source, carries an explicit
identity evidence value, and declares how byte identity was handled:

- `full_sha256` hashes a bounded regular file in full;
- `source_cas` reuses an existing content-addressed source identity;
- `image_reference` preserves a container image reference; and
- `not_hashed` makes the absence of a byte digest explicit.

The digest strategy and `Evidence<T>` status are separate. For example, a
large dataset can be intentionally un-hashed without being misrepresented as a
successfully content-hashed input. Materialized paths are optional metadata,
not portable identities.

## On-disk ownership

Evidence is compose metadata, not compute-node runtime state and not shared
cache content. For a validated scheduler job ID, the conceptual layout is:

```text
<compose-file-dir>/.hpc-compose/
└── evidence/
    └── <job-id>/
        ├── manifest.json
        ├── inputs.lock.json
        ├── events.jsonl
        ├── view.json
        └── events.lock
```

The submission-side writer owns `manifest.json` and `inputs.lock.json`. After
creation, no command may rewrite them in place. Annotation writers own event
appends. `view.json` has no independent authority: it is a cache derived from
the manifest and valid event prefix, and may be regenerated. The input lock is
validated independently against the exact digest pinned by the manifest.

`events.lock` is a protocol coordination file, not a disposable temporary
file. Keeping its inode stable allows independent processes to serialize
against the same lock even across repeated operations. Code must not remove and
recreate it while the evidence directory is live or as part of view repair;
whole-run cleanup removes the complete per-job evidence directory.

The evidence directory does not own scheduler state, logs, exported artifacts,
or the legacy `jobs/<job-id>.json` record. Deleting or repairing a materialized
view must never delete those other roots.

## Protocol invariants

Contributors must preserve these invariants:

1. `RunManifest` and `InputsLock` are immutable after successful publication.
2. Every document declares a schema version; readers reject unsupported future
   versions instead of guessing their meaning.
3. `run_id` is generated once and never reconstructed from `job_id`, a path,
   timestamp, or mutable configuration.
4. A run directory and its legacy submission record refer to the same validated
   scheduler job ID and immutable-record identity. Bundle export rejects a
   mismatched identity instead of rebinding stale metadata after scheduler-ID
   reuse.
5. Event writes hold the strict event-stream lock for the complete
   read/validate/append/view-refresh transaction.
6. Each committed event is one complete JSON object followed by a newline.
   Atomic publication preserves the prior complete stream as an identical
   prefix; existing events are never edited or reordered logically.
7. Event sequence numbers are monotonic within a run and are assigned while the
   lock is held. Duplicate or regressing sequence numbers are invalid.
8. `RunView` is a deterministic fold over the immutable manifest plus the valid
   event sequence. Rebuilding it produces the same logical value.
9. A view is published by atomic replacement; readers see either the previous
   complete view or the next complete view, never a partially written file.
10. Scheduler reads and other observation-only commands do not mutate evidence.

## Crash and concurrency behavior

The protocol favors recoverability over pretending that several files form a
filesystem transaction:

- Immutable documents are written completely to a private staging file in the
  same directory and atomically published. An existing document is not
  overwritten to repair a conflict.
- Event writers acquire the strict persistent lock before assigning sequence
  state or appending. A process that cannot acquire or validate the lock fails
  the mutation instead of performing an unlocked best effort.
- Events are serialized onto the existing complete byte stream and the enlarged
  stream is atomically published. If a process stops before refreshing
  `view.json`, the next annotation writer folds the event stream and republishes
  the view. Bundle export independently rebuilds its exported view in memory;
  this phase does not expose a user-facing repair command.
- A trailing partial JSONL record caused by a host or filesystem failure is not
  interpreted as an event. The current reader and writers fail closed, preserve
  the bytes for diagnosis, and block further appends. A future explicit repair
  must preserve the valid prefix and report the damaged suffix; it must not
  silently manufacture or reorder history.
- Concurrent annotation processes serialize on the same lock, so neither a tag
  update nor a note append can lose an already committed update.
- Whole-run cleanup takes the record lock and the existing evidence lock before
  deleting owned paths. It validates immediate managed parents without
  following symlinks and keeps the canonical record until payload deletion
  succeeds, so a failed cleanup remains safely retryable.
- A corrupt or missing view does not invalidate the immutable manifest or event
  history. The next annotation writer republishes it atomically without adding
  a repair event, while bundle export derives its own view without writing to
  tracked state and warns when validation fails. Commands that do not yet
  consume evidence continue to use legacy tracked state without hidden writes.

The shared-filesystem guarantees available on a cluster still matter. Atomic
rename and advisory locking must be supported coherently within the metadata
root. The protocol does not claim distributed consensus across unrelated
mounts, machines with incoherent locking, or manually copied partial trees.

These validation and append rules govern an individual evidence mutation. They
do not override the phase-one commit policy: once the compatible submission
record has been persisted, an evidence-layer error is reported as
`run_evidence_degraded` and the tracked operation remains successful.

## Privacy boundary

Run evidence is **not automatically safe to publish**. It can contain project
paths, scheduler identifiers, user-authored notes and tags, source references,
and other operational metadata. On Unix, hpc-compose restricts evidence
directories to the owner and creates evidence data with owner-only permissions.
Those permissions and the metadata root's access policy are the first privacy
boundary, not an export/redaction policy.

Writers must not persist secret values, expanded secret-file contents, tokens,
credentials, or full process environments. When identity is needed, prefer an
explicit non-secret identifier or digest with a documented scope. A digest is
still metadata and may reveal equality across runs; it is not a substitute for
an export/redaction policy.

No command should infer that copying the evidence directory creates a redacted
support bundle. A future exporter must define an allowlist, schema, and secret
canary tests independently.

## Compatibility and migration

The rollout is additive:

- Existing `SubmissionRecord` files remain readable and authoritative for the
  scheduler operations and phase-one record annotations that already use them.
- A legacy record without an evidence directory follows the legacy fallback.
  Read-only commands do not create evidence as a side effect.
- New submissions attempt to create evidence while retaining their compatible
  submission record, allowing old and new command paths to coexist during
  migration. Evidence failure produces `run_evidence_degraded`; it does not
  roll back or misreport an already-persisted tracked operation.
- Missing evidence is distinct from corrupt evidence. Readers should report or
  encode that distinction instead of mapping both cases to an empty object.
- Unknown future schema versions fail closed for typed interpretation. Their raw
  files are preserved so a newer binary can read them.
- Schema evolution is additive where possible. A change that alters identity,
  ordering, immutability, or evidence-state semantics requires a new schema
  version and explicit migration policy.
- Migration tools, if introduced, must be explicit mutations. They must not run
  implicitly from `status`, `ps`, `watch`, `replay`, or other reads.

The legacy fallback is a compatibility bridge, not permission to synthesize a
historically precise manifest from current compose files. If submit-time facts
are unavailable, their evidence status must say so.

## Verification expectations

Changes to the protocol should include focused tests for the affected invariant
and, when applicable, integration coverage through the command boundary. At a
minimum, maintain coverage for:

- schema-version validation and serialization round trips;
- globally unique opaque run IDs that are independent of scheduler job IDs;
- immutable-document conflict behavior;
- multiple concurrent appenders with no lost, duplicate, or malformed events;
- interrupted append handling and preservation of the valid JSONL prefix;
- deterministic rebuild after a missing or corrupt view;
- atomic view publication;
- legacy records with no evidence directory; and
- digest or modification-time checks proving scheduler reads are read-only.

Filesystem tests should use the same secure I/O, atomic-publication, and strict
locking helpers as production code. A test that only exercises an in-memory
fold does not verify the crash and concurrency contract.

## Contributor guidance

- Put new submit-time facts in the immutable manifest or input lock only when
  their meaning is stable at submission.
- Model post-submission user mutations as typed events; do not add another
  independently mutable JSON document.
- Treat the view as a query optimization. Do not make an irreversible decision
  depend on data that exists only in `view.json`.
- Keep command handlers thin: protocol validation, locking, folding, and atomic
  writes belong in the evidence module rather than being reimplemented by each
  CLI command.
- Do not append scheduler observations from polling reads. If scheduler events
  become desirable later, define an explicit writer and idempotency contract.
